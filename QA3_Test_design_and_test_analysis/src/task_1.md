# Техника тест-дизайна: Классы эквивалентности

**Объект:** Поле ввода логина на ресурсе по выдаче кредитов.
**Условие:** Поле принимает любые символы (длина не ограничена).

| Название класса эквивалентности | Описание класса | Тестовые значения (Data) |
| :--- | :--- | :--- |
| **Латинские буквы и цифры** | Стандартный набор символов для большинства систем авторизации. | `User777`, `admin` |
| **Специальные символы** | Проверка того, как система обрабатывает служебные знаки и знаки пунктуации. | `!@#$%^`, `(test)*` |
| **Кириллические символы** | Проверка поддержки кодировки (Unicode), отличной от латиницы. | `ИванИванович`, `Логин123` |
| **Пробельные символы** | Проверка обработки пробелов (в начале, в конце или внутри строки). | ` user `, `Ivan Ivanov` |
| **Математические символы и иероглифы** | Специфические символы для проверки устойчивости системы к разным шрифтам/языкам. | `∑∞π`, `你好` |

---

### Пояснение к выбору:
1. **Латиница и цифры**: Самый ожидаемый тип данных. Если система работает с ними, это базовая проверка работоспособности.
2. **Специальные символы**: Важны для безопасности. Некоторые системы могут некорректно обрабатывать такие символы, что иногда приводит к уязвимостям (например, SQL-инъекциям).
3. **Кириллица**: Поскольку ресурс, скорее всего, ориентирован на русскоговорящий сегмент (выдача кредитов), крайне важно проверить, не «сломается» ли база данных при вводе кириллических имен.
4. **Пробельные символы**: Часто разработчики забывают настроить автоматическую обрезку пробелов (trimming), из-за чего пользователь может не войти в систему, случайно поставив пробел в конце логина.
